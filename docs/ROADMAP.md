# Roadmap

Two phases. Both are now **unblocked**: the file format is decoded/validated and the device USB transfer protocol is fully reverse-engineered and hardware-validated. What's left in each is product engineering, not reverse engineering.

> **Adjacent scope:** reading *other Nord instruments'* program files is a
> bounded, hardware-free expansion on the shared CBIN container. **Shipped:** the
> `clavia/` container + `parseClaviaFile` model-codec registry, recognition +
> structure view, and **body decoders for Stage 2 & 3 plus a growing set of other
> models** (Electro, Piano, Lead, Wave, Grand, ‚Ä¶ ‚Äî ~14 models read today, more
> inferred via the shared container; see the in-app compatibility matrix and
> `docs/MULTI-MODEL.md`). Device transfer for non-Stage-4 models stays out of
> scope ‚Äî the transport is shared line-wide, but only Stage 4 is
> hardware-validated.

## Phase 1 ‚Äî Read & share (no hardware needed)

Everything here uses the file format, which is decoded and validated against ns4decode.

- [x] **Offset map ‚Äî DONE.** The full bit-location map for all four engines (406 params) is ingested from ns4decode's bitmaps (`offset-map.generated.ts`) and validated against the real fixture (80.6% byte coverage; bank/checksum/synth-layers correct). The Decode Inspector reads it now.
- [~] **Interpretation layer ‚Äî nearly complete; remaining gap now precisely characterized.** Sources validated against the fixture (0 mismatch):
  - **Pure params (202):** exhaustively evaluated from ns4decode's real interpreter (`values.generated.ts`) ‚Äî volume‚ÜídB, pan, pitch, envelope amounts, transpose, all simple enums.
  - **Morphs (94):** "X with wheel/A.T./pedal" resolved two-pass from their base value (`morphs.generated.ts`) ‚Äî `base(clamp(base+raw‚àí127))`, "none" at raw 127.
  - **Dependency-aware params:** timbre (piano-model), drawbars (organ-model), KB zones, filter-res, arp/LFO/FX rates, osc cat/wave (osc-type) ‚Äî wired via `deps.generated.ts` + `coverage.ts:depDisplay` (sibling-aware two-pass).
  - **Measured remaining gap (much smaller than the old "~110" estimate).** Decoding the fixture, only ~21 params render as raw, and most are *correctly* raw ‚Äî checksum, extern CC values, extern program, and genuinely integer-valued fields (drawbars, unison level, filter drive, amp-env velocity ‚Äî these match the ns4decode CSVs as plain ints). What was actually still unresolved:
    - **synth `analog cat`/`wave` for the ANALOG osc type ‚Äî DONE.** `deps.generated` only had FM-H/FM-I/WAVE; the ANALOG (type 0) branch is now ported from ns4decode (`synth-analog.generated.ts`, regenerated by `scripts/gen_synth_analog.py`), validated against the fixture (`4 (SHAPE)` / `2 (Shape Saw)`). Note: uses the current-firmware tables; pre-"SynthTypes" files used the `‚Ä¶Old` tables.
    - **Program-category name (CBIN 0x10) ‚Äî DONE.** ns4decode does **not** decode this header byte, so the enum was sourced instead from Nord Sound Manager's authoritative 54-entry table (`S_Category_GetName`, see `docs/NSM-TEARDOWN.md`) and cross-checked against a real Stage 4 corpus over USB. Ported in `categories.ts` (e.g. `6 = Lead`, `7 = Organ`, `45 = Synth Classic`). Values 47+ are unnamed in firmware 3.40 (reserved); the raw id is surfaced for those.
- [x] **Map raw ‚Üí `NS4Program`** ‚Äî DONE. `parseNs4Program` classifies the file and decodes it into the `NS4Program` model (name/category, engine layers, the `Morphable` system, sample refs); `src/lib/ns4/parse.ts`.
- [x] **Visualize a program** ‚Äî DONE. A readable program view (`src/components/program/`): per-engine cards (piano/sample, organ, synth), effects, the morph system, key zones, and referenced samples ‚Äî for **Stage 2 / 3 / 4** (`Ns2ProgramView`/`Ns3ProgramView` + the shared `ProgramView`/`DecodedProgramView`).
- [x] **Browse a library** ‚Äî DONE. A mobile-first **Library** home ‚Äî folder scan + single-file import (IndexedDB-persisted), split into **Programs ¬∑ Samples ¬∑ Presets** with a master/detail view ‚Äî plus a line-wide **compatibility matrix**.
- [x] **Sample workshop** ‚Äî DONE. Inspect `.nsmp`/`.nsmp3`/`.nsmp4` (keyboard map, loops, root notes); a **playable on-screen keyboard + MIDI-controller rompler** (pitched, polyphonic, velocity + sustain); convert across generations; WAV import + per-sample / export-all-to-zip; persistent sample import.
- [~] **AI search** over parsed programs ("punchy clav", "ambient pad") ‚Äî a zero-config keyword ranker ships (`lib/ai/search.ts`); the LLM ranker is still stubbed (provider-pluggable, Claude by default).
- [ ] **AI explain** ‚Äî "what does this patch do, and how would I tweak it?"
- [ ] **Community library** (backend) ‚Äî upload, browse, search, rate, fork user programs. *Programs only, never samples (`docs/LEGAL.md`).*
- [x] **Sample resolution ‚Äî "you need these" ‚Üí official downloads.** *(Done ‚Äî Slice 4: `resolveFactory` + the catalog snapshot; see `docs/FACTORY-LIBRARIES.md`.)* A shared program lists the factory samples it references (`programSampleRefs()`). OpenNord shows that list and, for missing ones, **deep-links to Nord's free [Sample Library](https://www.nordkeyboards.com/sounds/sample-library/)** (free, organized by category) so the user downloads the audio from the official source. *(The official client resolves factory content from a fetchable S3 manifest ‚Äî `clavia_sound_libraries.xml`, see `docs/NSM-TEARDOWN.md` ‚Äî a candidate canonical id‚Üídownload source.)* OpenNord **never hosts or transfers sample audio** ‚Äî it only recognizes a sample (`sample.ts`, header-only), names it, and points to where to get it. Needs: the sample-id‚Üíname/category map (the hard-tail item above) and, ideally, confirming the id‚Üîfile-CRC linkage so "missing" detection is exact. User-created samples are excluded (they have no official URL ‚Äî they're the user's own, sharable file).
- [ ] **AI generate** ‚Äî "warm Rhodes with slow tremolo" ‚Üí a `.ns4p` you can download.

A useful product exists at the end of Phase 1 *even if the keyboard is never touched.*

## Phase 2 ‚Äî Talk to the Nord (was the frontier ‚Äî now solved)

The device-transfer protocol is **fully reverse-engineered and hardware-validated**, and a WebUSB desktop client is built (`src/lib/device/`). What's left here is product UI and broader runtimes (Electron/node-usb, native iPad), not RE.

- [x] **Transfer protocol ‚Äî SOLVED & hardware-validated, both directions.** Vendor **raw-USB bulk protocol** on interface 0 (`0x03`/`0x82`/`0x81`), recovered from the binary and **proven live on a Stage 4**: framing `[len][protoId 0x0C][ver 0x0A][msgId][payload][CRC16]` (big-endian), full opcode table, unified `{bank,slot}` addressing in a `Begin{partition}` session. Validated end-to-end: **enumerate** (FileIterate walk ‚Äî listed all 356 files), **read** (pulled real program bodies), and **write** (created a playable program at C:88). Full spec + `scripts/nord*.c` tools: `docs/PROTOCOL-RE.md`. (Transport-agnostic in code ‚Äî a Clavia SysEx framing exists ‚Äî but the NS4 doesn't service SysEx transfer, so it's vendor-USB only: desktop + native iPad M1+, not iPhone/PWA; `docs/SYSEX-SPIKE.md`.)
- [x] **Pull/push programs ‚Äî WebUSB runtime built.** A desktop (Chromium **WebUSB**) client is implemented in `src/lib/device/` (`transport`/`webusb` ‚Üí `session` ‚Üí `transfer`), with windowed **multi-chunk** read/write (`pullFile`/`pushFile`) and a `device.tsx` route. Remaining runtimes: an Electron/**node-usb** wrapper and a **native iPad app (M1+) with a USBDriverKit DEXT**. Not on iPhone or a PWA (no vendor-USB reach there).
- [x] **Push a patch** to the keyboard ‚Äî write opcodes (`FileCreate 0x0A`, `FileWrite 0x10`, `FileClose 0x0E` to commit) are **validated live** (a playable program written to C:88; see `docs/PROTOCOL-RE.md`) and implemented in `transfer.ts:pushFile`. Still: target a throwaway slot and back up first (`FileClose` is what commits ‚Äî without it the device silently discards).
- [x] **Live tweak** ‚Äî real-time CC/NRPN control (the ns4mcp parameter map already exists; this is the easy, proven part of device comms).
- [x] **Back up & reorganize** ‚Äî an offline **backup organizer** loads and re-exports multi-GB `.ns4b` backups (streamed, dodging the 2 GiB `arrayBuffer` wall), and a **reorg/move engine** repositions programs/samples with a copy‚Üíverify‚Üídelete journal + rollback (the device has no native move opcode). `src/lib/device/`.

## What's proven vs. unproven

| Capability | State |
|---|---|
| Parse `.ns4p` (partial) | **Proven** (ns4decode) ‚Äî re-derive openly |
| Visualize parsed programs | **Built** ‚Äî readable program view, Stage 2/3/4 (`src/components/program/`); share + AI still to come |
| Real-time CC/NRPN control | **Proven** (ns4mcp / standard MIDI) |
| **Read** programs/files off the Stage 4 over USB | **Proven** ‚Äî protocol RE'd + validated live (`docs/PROTOCOL-RE.md`) |
| **Write** programs to the Stage 4 | **Proven** ‚Äî written live to C:88; implemented in `src/lib/device/transfer.ts` (`pushFile`). Target a spare slot + back up first |
| Program transfer reach | **Desktop** (WebUSB/node-usb) + **native iPad M1+** (USBDriverKit DEXT, vendor-class allowed). **Not** iPhone or any PWA; **not** SysEx (NS4 doesn't do it). See `docs/SYSEX-SPIKE.md` |

The RE is effectively complete: `.ns4p` decode + interpretation are validated 0-mismatch against ns4decode; checksum (CRC-32), the write path (`writer.ts`), bundles, and sample headers are implemented and tested; and the full USB transfer protocol (enumerate/read/write) is proven on hardware **and** built as a WebUSB client (`src/lib/device/`). The remaining frontier is the **product UI** (visualize / AI / community library) and broader transfer-runtime packaging (Electron/node-usb, native iPad).
